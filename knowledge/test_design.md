# 算法测试用例设计

## 为什么要设计测试用例
算法代码生成后必须进行验证。测试用例可以帮助判断代码是否真正符合需求，而不是只看代码表面是否完整。

## 测试类型

### 正常情况
输入符合题目要求，结果应正确。

示例：
```text
binary_search([1, 3, 5, 7], 5) -> 2
```

### 边界情况
输入规模很小或处于边界位置。

示例：
```text
binary_search([], 1) -> -1
binary_search([1], 1) -> 0
```

### 特殊情况
包含重复元素、已排序、逆序、全部相同等。

示例：
```text
bubble_sort([2, 2, 1]) -> [1, 2, 2]
```

### 异常情况
输入不符合预期，需要程序给出提示或合理处理。

示例：
```text
Dijkstra 输入负权边，应提示不适用。
```

## 测试表建议字段
- 测试编号
- 输入内容
- 预期输出
- 实际输出
- 是否通过
- 失败原因或改进建议

## 判断标准
- 输出是否与预期一致。
- 是否覆盖边界输入。
- 是否能处理错误输入。
- 代码是否能直接运行。
- 结果说明是否适合初学者理解。

## 报告写法建议
报告中不要只写“测试通过”，应至少展示若干典型输入、模型输出、人工评价和失败案例。失败案例可以体现对系统局限性的认识。
